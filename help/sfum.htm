1     Introduction

1.1         Project Context

Street Factory application allows fully automatic processing of any oblique and street imagery for the generation of 3D urban models.

This application is a fully integrated solution with software and hardware allowing high productivity and fast, automated processing of imagery in order to manufacture 3D digital urban model as 3D Textured Triangle Irregular Network.

In order to provide such efficiency, quality and automation, Street Factory is designed around four main axes:

  • precise and robust geometry management,
  • effective photogrammetric measurement from any sensors,
  • 3D automated textured TIN generation from multiple views,
  • open output format for easy integration with OpenSceneGraph.

Moreover the system uses parallel GPU server processing efficiency in order to deliver fast productivity even with thousands of images.

Street Factory runs on a parallel distributed processing system. All the computation work is done on a server and compute nodes. The server is the machine that provides a variety of services that are needed by the system including database maintenance and dispatching computing jobs to the nodes. Computing nodes are the machines used mainly for computation. Nodes receive the computation jobs from the server.

This project consists in the development of the StreetFactory client part.

 

1.2         Purpose of the document

This document is the Users Manual (UM) of the StreetFactory client part.

 

1.3         Document overview

The document is the Software Design Document. It is composed with the following parts:

  • the chapter 1 introduces the document,
  • the chapter 2 gives an overview of StreetFactory client graphical interface,
  • the chapter 2.3 details the different StreetFactory client perspectives,
  • the chapter 4 presents the external tools,
  • the chapter 5 describers the other features of StreetFactory client.

Street Factory is a fully integrated solution with software and hardware allowing high productivity and fast, automated processing of imagery in order to manufacture 3D digital urban model as 3D Textured Triangle Irregular Network (TIN).

·        precise and robust geometry management

·        effective photogrammetric measurement from any sensors

·        3D automated textured TIN generation from multiple views

·        open output format for easy integration with OpenSceneGraph.

Moreover the system uses parallel GPU server processing efficiency in order to deliver fast productivity even with thousands of images.

Street Factory runs on a parallel distributed processing system. All the computation work is done on a server and compute nodes. 3.4.1    Jobs view

The Jobs view allows to visualize the state of the workflows and activities running for the open project:

 

Figure 28 : Jobs view

As soon as a workflow or a standalone activity is created from the Workflows and Activities perspective, it appears in the Jobs view.

For each workflow or a standalone activity, the following settings are displayed:

·        Name: name of the workflow or the activity

·        Status: status of the workflow or the activity which can be:

o   CREATED: the workflow or the activity has been created from the Workflows and Activities perspective,

o   SUBMITTED: the workflow or the activity has been submitted to the StreetFactory server,

o   STARTED: the workflow or the activity has been started by the StreetFactory server,

o   RUNNING: the workflow or the activity is running at the StreetFactory server side,

o   STOPPED (only available for a workflow): the workflow has been stopped by a breakpoint,

o   CANCELED: the workflow or the activity has been manually cancelled  by the user,

o   WARNING: the workflow or the activity execution is correctly finished but has thrown warnings),

o   FAILED: the workflow or the activity execution has failed,

o   SUCCESSFUL: the workflow or the activity execution is correctly finished without warning.

·        Progress: workflow or the activity progress percentage,

·        Start time: workflow or the activity start time,

·        End time: workflow or the activity end time

·        Duration: workflow or the activity duration,

·        Owner: user which has been submitted the workflow or the activity

·        Id: workflow or the activity identifier,

·        Execution host: host  on which runs the workflow or the activity.

For each workflow, the child activities are displayed with the associated status.

For ach activity, the child jobs are listed with the associated status.

 

The Jobs view offers the following filter capabilities:

Allows to show/hide the workflows

Allows to show/hide the standalone activities

Allows to show/hide the workflows and activities having the CREATED status

Allows to show/hide the workflows and activities having the RUNNING status

Allows to show/hide the workflows and activities having the STOPPED status

Allows to show/hide the workflows and activities having the SUCCESSFUL status

Allows to show/hide the workflows and activities having the WARNNING status

Allows to show/hide the workflows and activities having the FAILED status

Allows to show/hide the workflows and activities which belong to the current logged user

Allows to show/hide the workflows and activities which do not belong to the current logged user

Allows to filter using a Java regular expression
(see appendix 1 for Java regular expression description)

 

In addition, the Jobs view offers the following buttons:

Allows to start the execution of the selected workflow or activity
(Available only if the selected workflow or activity has one of the following status: CREATED, CANCELLED, STOPPED, FAILED)

Allows to cancel the execution of the selected workflow or activity
(Available only if the selected workflow or activity has the RUNNING status)

Allows to remove the execution of the selected workflow or activity
(Available only if the selected workflow or activity has the CREATED status)

APPENDIX 1: Java regular expressions

 

A regular expression defines a search pattern for strings. The search pattern can be anything from a simple character, a fixed string or a complex expression containing special characters describing the pattern. The pattern defined by the regular expression may match one or several times or not at all for a given string.

A simple example for a regular expression is a (literal) string. For example, the Hello World regular expression will match the "Hello World" string. . (dot) is another example for a regular expression. A dot matches any single character; it would match, for example, "a" or "z" or "1".

The following tables lists several example regular expressions and describes which pattern they would match.

 

Regular Expression

Matches

this is text

Matches exactly "this is text"

this\s+is\s+text

Matches the word "this" followed by one or more whitespace characters followed by the word "is" followed by one or more whitespace characters followed by the word "text".

^\d+(\.\d+)?

^ defines that the patter must start at beginning of a new line. \d+ matches one or several digits. The ? makes the statement in brackets optional. \. matches ".", parentheses are used for grouping. Matches for example "5", "1.5" and "2.21".

 

The following description is an overview of available meta characters which can be used in regular expressions. This chapter is supposed to be a references for the different regular expression elements.

 

Regular Expression

Description

.

Matches any character

^regex

Finds regex that must match at the beginning of the line.

regex$

Finds regex that must match at the end of the line.

[abc]

Set definition, can match the letter a or b or c.

[abc][vz]

Set definition, can match a or b or c followed by either v or z.

[^abc]

When a caret appears as the first character inside square brackets, it negates the pattern. This ccontent/an match any character except a or b or c.

[a-d1-7]

Ranges: matches a letter between a and d and figures from 1 to 7, but not d1.

X|Z

Finds X or Z.

XZ

Finds X directly followed by Z.

$

Checks if a line end follows.

 

 

 

The following meta characters have a pre-defined meaning and make certain common patterns easier to use, e.g., \d instead of [0..9].

 

Regular Expression

Description

\d

Any digit, short for [0-9]

\D

A non-digit, short for [^0-9]

\s

A whitespace character, short for [ \t\n\x0b\r\f]

\S

A non-whitespace character, short for [^\s]

\w

A word character, short for [a-zA-Z_0-9]

\W

A non-word character [^\w]

\S+

Several non-whitespace characters

\b

Matches a word boundary where a word character is [a-zA-Z0-9_].

 

A quantifier defines how often an element can occur. The symbols ?, *, + and {} define the quantity of the regular expressions:

 

Regular Expression

Description

Examples

*

Occurs zero or more times, is short for {0,}

X* finds no or several letter X,
.* finds any character sequence

+

Occurs one or more times, is short for {1,}

X+ - Finds one or several letter X

?

Occurs no or one times, ? is short for {0,1}.

X? finds no or exactly one letter X

{X}

Occurs X number of times, {} describes the order of the preceding liberal

\d{3} searches for three digits, .{10} for any character sequence of length 10.

{X,Y}

Occurs between X and Y times,

\d{1,4} means \d must occur at least once and at a maximum of four.

*?

? after a quantifier makes it a reluctant quantifier. It tries to find the smallest match. This makes the regular expression stop at the first match.
